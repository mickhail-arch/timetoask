UX_SCENARIOS.md
Сценарии и состояния — как пользователь движется по продукту
Продукт: Таймтуаск  ·  app.site.ru  ·  Версия: 1.0 MVP

Назначение документа
Описывает пользовательские сценарии: последовательность шагов, переходы между экранами, поведение системы в каждой точке. Cursor читает этот файл при реализации логики переходов и обработки состояний.

•	Каждый сценарий = таблица шагов: экран → действие → реакция системы
•	Описаны все ветки: happy path + error path + edge cases
•	UX-решения обоснованы через логику бэкенда — ничего не противоречит архитектуре

1. Вход и регистрация / Auth Flow
Все auth-страницы — без оболочки кабинета. Middleware перехватывает незащищённые маршруты.

1.1 Регистрация — happy path
Шаг	Экран / URL	Действие	Реакция системы
1	/register	Вводит email, нажимает «Регистрация»	POST /api/auth/register → отправляет код на email. Редирект на /register/verify
2	/register/verify	Вводит код из письма, нажимает «Подтвердить»	POST /api/auth/verify-email → код верный. Редирект на /register/password
3	/register/password	Вводит пароль + подтверждение, нажимает «Завершить регистрацию»	POST /api/auth/reset-password/confirm → аккаунт создан, сессия открыта, начислены FREE_TOKENS_ON_REGISTER. Редирект на /dashboard

1.2 Регистрация — error paths
Ситуация	Где	Реакция UI
Email уже занят	/register	Ошибка под полем: «Аккаунт с таким email уже существует»
Невалидный email	/register	Ошибка под полем: «Введите корректный email»
Неверный код	/register/verify	Border error на поле + текст «Неверный код»
Код истёк (>15 мин)	/register/verify	Ошибка + ссылка «Отправить повторно» становится активной
Пароль слишком простой	/register/password	Ошибка: «Используйте не менее 8 символов, включая буквы и цифры»
Пароли не совпадают	/register/password	Ошибка: «Пароли не совпадают. Проверьте введённые данные»

1.3 Вход — happy path
Шаг	Экран / URL	Действие	Реакция системы
1	/login	Вводит email + пароль, нажимает «Войти»	POST /api/auth/login → сессия создана (httpOnly cookie). Редирект на /dashboard

1.4 Вход — error paths
Ситуация	Реакция UI
Неверный email или пароль	Ошибка под полем пароля: «Неверный email или пароль». Оба поля с border error
Аккаунт заблокирован (V2)	Toast error: «Аккаунт заблокирован. Обратитесь в поддержку»
Сетевая ошибка	Toast error: «Ошибка соединения. Попробуйте ещё раз»

1.5 Сброс пароля
Шаг	Экран / URL	Действие	Реакция системы
1	/reset-password	Вводит email, нажимает «Отправить код»	POST /api/auth/reset-password → код отправлен. Редирект на /reset-password/verify
2	/reset-password/verify	Вводит код из письма	POST /api/auth/verify-email → верный. Редирект на /reset-password/confirm
3	/reset-password/confirm	Вводит новый пароль + подтверждение	POST /api/auth/reset-password/confirm → пароль изменён. Редирект на /login с Toast: «Пароль успешно изменён»

1.6 Защита маршрутов
Сценарий	Поведение
Незарегистрированный открывает /dashboard	middleware.ts → редирект на /login
Авторизованный открывает /login или /register	middleware.ts → редирект на /dashboard
Не-admin открывает /admin	middleware.ts → редирект на /dashboard
Сессия истекла в процессе работы	Следующий API-запрос → 401 → редирект на /login с Toast: «Сессия истекла»

2. Первый запуск / First Run
Сценарий нового пользователя — от регистрации до первого результата.

Шаг	Экран / URL	Действие	Реакция системы
1	/dashboard	Попадает после регистрации	Видит каталог инструментов. Баланс в хедере: FREE_TOKENS_ON_REGISTER тк
2	/dashboard	Кликает на карточку инструмента	Переход на /tools/:id. Breadcrumb обновляется
3	/tools/:id	Видит форму, заполняет поля	AutoForm сгенерирована из schema.ts плагина
4	/tools/:id	Нажимает кнопку «Запустить бесплатно»	Кнопка показывает сколько бесплатных запусков осталось
5	/tools/:id	Ждёт результат	Sync: текст стримится. Async: прогресс по шагам
6	/tools/:id	Получает результат	Кнопки «Копировать» и «Новый запрос». Баланс обновился

→ FREE_TOKENS_ON_REGISTER начисляются автоматически при создании аккаунта (instrumentation.ts)
→ free_uses_limit — у каждого инструмента свой. После исчерпания — списание токенов

3. Работа с инструментом / Tool Execution Flow

3.1 Sync-инструмент (SSE, < 30 сек)
Шаг	Экран / URL	Действие	Реакция системы
1	/tools/:id	Заполняет форму, нажимает «Запустить»	AutoForm валидирует данные через Zod. При ошибке — подсветка полей
2	/tools/:id	—	reserve() → блокирует токены. POST /api/tools/:id/execute → открывает SSE-соединение
3	/tools/:id	Наблюдает за стримингом текста	StreamResult накапливает content. Курсор мигает. Форма заблокирована
4	/tools/:id	—	SSE закрывается. finalize() → списывает токены. usage_log записан
5	/tools/:id	Копирует результат или запускает снова	Кнопки «Копировать» и «Новый запрос» активны. Баланс в хедере обновился

3.2 Async-инструмент (jobId + polling, > 30 сек)
Пример: SEO-генератор статей — 10 шагов пайплайна.
Шаг	Экран / URL	Действие	Реакция системы
1	/tools/:id	Заполняет форму, нажимает «Запустить»	Валидация Zod. reserve() токены. POST /api/tools/:id/execute → возвращает { jobId }
2	/tools/:id	Видит JobProgress с шагами	Клиент polling GET /api/jobs/:jobId/status каждые 2 сек
3	/tools/:id	Наблюдает прогресс по шагам	Каждый шаг: pending → active → done. Текущий шаг выделен. Процент выполнения
4	/tools/:id	—	Последний шаг завершён. status=completed. finalize() → списание. usage_log записан
5	/tools/:id	Видит полный результат	JobProgress скрывается. JobResult показан (HTML статья + мета-теги + SEO-отчёт)
6	/tools/:id	Скачивает или копирует	Кнопки по output_format: «Копировать HTML», «Скачать», «Новый запрос»

3.3 Ошибки выполнения
Ситуация	Поведение системы	Реакция UI
Ошибка LLM (OpenRouter)	rollback() → токены возвращены	Toast error: «Ошибка генерации. Токены не списаны»
Таймаут (> MAX_TIMEOUT)	rollback() → токены возвращены	Toast error: «Превышено время ожидания. Попробуйте ещё раз»
Ошибка сети на клиенте	SSE разорвано, polling упал	Toast error + кнопка «Повторить» (повторно открывает polling для async)
job застрял (stale job)	cleanupStaleJobs при старте	При следующем polling: status=failed → UI показывает ошибку
→ Во всех ошибочных случаях форма разблокируется — пользователь может попробовать снова

4. Баланс и пополнение / Billing Flow

4.1 Пополнение баланса — happy path
Шаг	Экран / URL	Действие	Реакция системы
1	/billing	Выбирает пресет (напр. 1000 тк), нажимает «Пополнить»	POST /api/billing/checkout → создаёт платёж в ЮKassa. Редирект на страницу оплаты ЮKassa
2	ЮKassa (внешняя)	Вводит данные карты, подтверждает платёж	ЮKassa обрабатывает платёж
3	/billing	Возвращается на /billing	GET /api/billing/balance показывает обновлённый баланс. Toast: «Баланс пополнен на N тк»
→ Webhook POST /api/billing/webhook (HMAC-SHA256) → обновляет баланс в БД асинхронно
→ YOKASSA_RETURN_URL = /billing — пользователь возвращается сюда после оплаты

4.2 Нулевой баланс — сценарий
Шаг	Экран / URL	Действие	Реакция системы
1	/tools/:id	Пытается запустить инструмент без баланса	AutoForm заблокирована. Кнопка: «Пополнить баланс» вместо «Запустить»
2	/tools/:id	Кликает «Пополнить баланс»	Переход на /billing
3	/billing	Пополняет баланс	Возвращается. Кнопка снова «Запустить»

4.3 Баланс заканчивается в процессе
Ситуация	Реакция
Баланс упал до 0 после запуска	BalanceWidget в хедере обновится (polling 30 сек). Следующий запуск → кнопка «Пополнить баланс»
reserve() не хватает токенов	API возвращает 402. Toast error: «Недостаточно токенов». Форма остаётся активной

4.4 Ошибки оплаты
Ситуация	Реакция UI
Платёж отклонён банком	ЮKassa возвращает на /billing. Toast error: «Платёж не прошёл. Попробуйте другую карту»
Webhook не пришёл (>5 мин)	Баланс не обновился. Пользователь видит старый баланс. Поддержка разбирается вручную (MVP)
Двойное нажатие «Пополнить»	idempotency_key защищает от двойного создания платежа

5. Пустые состояния и ошибки / Empty & Error States
Правила применяются ко всем страницам и блокам с данными.

5.1 Четыре обязательных состояния
Состояние	Когда	Что показывать
loading	Данные загружаются	Skeleton — точные заглушки по форме контента. Никаких спиннеров на всю страницу
empty	Данных нет (но это ок)	Иконка + короткий текст что делать. Кнопка действия если применимо
error	Запрос упал	Toast error + кнопка «Повторить» + текст ошибки если нужен
filled	Данные загружены	Контент

5.2 Тексты пустых состояний
Блок	Текст empty-состояния
TransactionHistory	Транзакций пока нет
LogsTable	Вы ещё не запускали инструменты
ToolsGrid	Инструменты появятся совсем скоро
AdminUsers	Пользователей пока нет
AdminLogs	Логов пока нет

5.3 Toast — правила
Тип	Цвет	Когда использовать	Длительность
success	Зелёный (#34C759)	Успешное действие: пароль изменён, баланс пополнен	4000ms
error	Красный (#FF3B30)	Ошибка: запрос упал, неверные данные, нет баланса	6000ms
info	Синий (#007AFF)	Нейтральная информация: код отправлен повторно	4000ms
→ Позиция: bottom-right. Максимум 3 toast одновременно — старые вытесняются
→ Toast не заменяет inline-ошибки полей — они работают вместе

5.4 Глобальные ошибки
Код	Страница	Что показать
404	app/not-found.tsx	Текст «Страница не найдена» + кнопка «На главную»
403	Inline на странице	Текст «Доступ запрещён» + кнопка «На Dashboard»
500	app/error.tsx	Текст «Что-то пошло не так» + кнопка «Попробовать снова»
401	Редирект	middleware.ts → /login с параметром ?from=текущий_путь

6. Роли / Roles
Две роли: user и admin. Роль хранится в БД (users.role) и передаётся в NextAuth session.

6.1 Различия в UI
Элемент UI	user	admin
Пункт «Админка» в UserMenu	Скрыт	Виден
Маршрут /admin	403	Доступен
Колонка «Стоимость $» в LogsTable	Скрыта	Видна
LogsTable	Только свои записи	Все пользователи
AdminTabs (Пользователи/Инструменты)	Нет	Есть

6.2 Как определяется роль в UI
•	Server Components — читают session на сервере: getServerSession(authOptions)
•	Client Components — читают через хук useSession() из NextAuth
•	Условный рендер: {session.user.role === "admin" && <AdminLink />}
•	middleware.ts — проверяет /admin маршруты до рендера
→ Роль никогда не хранится в localStorage или клиентском стейте — только из сессии

7. Навигация и переходы

7.1 Карта маршрутов
URL	Страница	Защита
/login	Вход	Публичный. Авторизованных → /dashboard
/register	Регистрация шаг 1	Публичный
/register/verify	Регистрация шаг 2	Публичный
/register/password	Регистрация шаг 3	Публичный
/reset-password	Сброс пароля	Публичный
/dashboard	Каталог инструментов	Требует сессии
/tools/:id	Страница инструмента	Требует сессии
/billing	Баланс и пополнение	Требует сессии
/logs	История использования	Требует сессии
/profile	Профиль	Требует сессии
/admin	Админка	Требует role=admin

7.2 Правила навигации
•	После любого действия с данными (сохранить, удалить) — Toast, не редирект
•	Редирект только при смене контекста: вход → /dashboard, выход → /login
•	Breadcrumb обновляется автоматически через usePathname() — вручную не задаётся
•	Активный пункт Sidebar — по точному совпадению pathname с href nav-пункта
•	После пополнения баланса — остаться на /billing, не редиректить
•	«Назад» в браузере — работает стандартно, никаких перехватов

Правила использования
•	Каждый сценарий покрывает happy path + все error paths
•	Ошибки API всегда показываются через Toast, inline-ошибки — только для полей форм
•	Skeleton при загрузке обязателен — пользователь видит форму контента, не пустоту
•	rollback() токенов при любой ошибке выполнения — пользователь не теряет деньги
•	Этот файл синхронизируется с: UI_PAGES.md, UI_COMPONENTS.md, FRONTEND_ONBOARDING.md
